# HOWTO — Build a new simulation

End-to-end recipe for adding a new (region, election) combination to Kaisim.

The example throughout uses a hypothetical `up_2027_ac042` (Uttar Pradesh AC #42)
to make naming concrete. Replace with your own.

---

## 0. Prerequisites

You should have:
  - Tabular demographic data for the target geography. Marginals (single-axis
    distributions) are required; conditional joints (cross-tabs) are highly
    desirable.
  - At least one historical election result you'd like to calibrate against.
  - Optionally: per-axis vote-share tables (P(party | religion) etc.).

If you only have marginals and no joints, the pipeline still works — the DAG
becomes a flat list of independent axes and verifier joint checks are sparse.

---

## 1. Scaffold the simulation folder

```
simulations/up_2027_ac042/
├── README.md
├── build_structures.py         # writes structures/{axes,joints}.json from your CSVs
├── structures/                 # generated by build_structures.py
│   ├── axes.json
│   └── joints.json
├── derived/                    # region-specific Python plugins
│   ├── __init__.py
│   └── (one file per derived axis or non-trivial sampling rule)
├── prompts/                    # Phase 1.2: LLM prompts
│   ├── system_anchors.md       narrative facts the LLM should know
│   ├── persona_rules.md        coherence rules
│   └── batch_guidance.md       gap-fill instructions template
├── news/                       # opinion / event context
│   ├── persona_context_pre<year>.md
│   └── timeline_<from>_<to>.jsonl       (Phase 2 drip-feed)
├── persona_configs/            # YAMLs — see step 5
├── personas/                   # generated artifacts (gitignored or LFS)
├── experiments/                # Phase 2 run configs
└── runs/                       # Phase 2 run outputs
```

Use `simulations/wb_2021_ac095/` as the reference; copy its layout.

---

## 2. Write `build_structures.py`

This script translates your raw CSVs into framework-consumable JSON. It is
the only place the simulation reads CSV.

Structure of `axes.json`:

```json
{
  "$schema_version": "1.0",
  "axes": [
    { "name": "<axis_name>",
      "kind": "partition" | "flag" | "scalar" | "derived",
      ...kind-specific fields,
      "tier": "A"|"B"|"C"|"D"|"E",
      "source_csv": "<filename>"
    }
  ]
}
```

Structure of `joints.json`:

```json
{
  "$schema_version": "1.0",
  "joints": [
    { "name": "<joint_name>",
      "parents": ["<parent_axis>", ...],
      "child":   "<child_axis>",
      "semantics": "conditional" | "flag_rate_conditional"
                 | "two_indicator_rates" | "vote_conditional",
      "table": { "<parent_value>": { "<child_label>": <pct>, ... } },
      "tier": "<A-E>",
      "source_csv": "<filename>",
      ...optional bucket maps and "use" flags
    }
  ],
  "aggregate_targets": [
    { "name": "vote_aggregate_<year>",
      "field": "vote_<year>",
      "buckets": [
        {"name": "<bucket>", "target_pct": <float>, "vote_values": ["<party_code>", ...]}
      ],
      "tier": "<A-E>",
      "source_csv": "<filename>"
    }
  ]
}
```

See [`HOWTO_AXES_AND_JOINTS.md`](HOWTO_AXES_AND_JOINTS.md) for the design
rules: kind selection, tier guidance, bucket-map usage, naming conventions.

Run `python simulations/up_2027_ac042/build_structures.py` and verify the
JSON outputs spot-check correctly (marginals sum to 100, joint table keys
match axis category codes, etc.).

---

## 3. Write derived plugins (region-specific code)

Anything that can't be expressed as a native sampling method is a plugin.
Common cases:

  - **Welfare/scheme exposure** — eligibility rules from prior fields.
  - **Workforce status** — age/gender hard rules combined with edu joint.
  - **Marital status** — currently_married joint + age-pattern residual.
  - **Asset blending** — multiple parent joints combined per flag.
  - **Conditional categorical with guards** — e.g., `occupation = "None"` if
    not a worker.

Plugin signature (single function in a `.py` file under `derived/`):

```python
def sample(rng, ctx, axis, persona_so_far, persona_config=None, **kwargs):
    """
    rng              random.Random — seeded by the orchestrator
    ctx              SimulationContext — use ctx.joint(name), ctx.axis(name)
    axis             the Axis being sampled
    persona_so_far   {axis_name: value} of fields already sampled
    persona_config   PersonaConfig — for reading config knobs
    kwargs           extras passed via sampling_spec[axis_name].kwargs
    return           the sampled value (str | bool | dict | None)
    """
```

Reference: [`simulations/wb_2021_ac095/derived/`](../simulations/wb_2021_ac095/derived/).

## 4. Write your first persona_config

Create `simulations/up_2027_ac042/persona_configs/baseline_rule.yaml`. The
schema is documented in
[`simulations/wb_2021_ac095/persona_configs/baseline_rule.yaml`](../simulations/wb_2021_ac095/persona_configs/baseline_rule.yaml)
— copy it and adapt the `sampling_spec`, `vote.blending.inputs`, and
`verifier.tolerance` blocks to your axes and tables.

Key choices to make:

  - **sampler**: `rule_based` for the free reference set, `llm_batch` for
    LLM-generated personas (Phase 1.2).
  - **target_n** + **batch_size**: 100/20 is the recommended starting point
    for LLM runs; for rule-based, set both to your full target.
  - **vote.blending.method**: `log_odds_mean` (conservative default) or
    `naive_bayes` (sharper, assumes axis-independence).
  - **vote.blending.inputs**: weights per axis. Higher weight → that axis
    contributes more to the blended vote distribution.
  - **vote.welfare_overlap.method**: `precedence` (pick first match),
    `mean` (average qualifying rows), `dominant` (max-spread row).
  - **verifier.tolerance**: tighten for n=1000+ runs.

---

## 5. Run the rule-based baseline

```bash
python simulations/up_2027_ac042/generate.py baseline_rule
```

This calls the framework, walks the DAG, applies your plugins, runs the
verifier, and writes everything to
`simulations/up_2027_ac042/personas/<set_name>/`:

```
personas/baseline_rule_n1000/
├── personas.jsonl              one Persona per line
├── meta.json                   config_hash, n, sampler, verifier headline
├── persona_config.snapshot.yaml
├── batches/                    (LLM samplers only — rule-based skips)
└── reports/
    └── FINAL.md                Markdown verifier report
```

Inspect `reports/FINAL.md`. You should see:
  - Marginals reasonably close to targets (esp. for axes with no plugin twist).
  - Joint cells with bucketed parents/children matching after IPF / bucket maps.
  - Aggregate vote shares within ~5pp of the calibration target.

If a marginal is wildly off, the relevant plugin's heuristic is likely
wrong. If a joint cell is off but the marginal is fine, the joint's
bucket-map is likely wrong. If the aggregate vote is off, your
`vote.blending.weights` need adjustment.

---

## 6. Iterate

Make a second `persona_config` (e.g., `default.yaml`) with tweaks:

```bash
cp simulations/up_2027_ac042/persona_configs/baseline_rule.yaml \
   simulations/up_2027_ac042/persona_configs/default.yaml
# edit default.yaml — set sampler: llm_batch, target_n: 100, batch_size: 20
```

Run again with the new config. The two persona sets coexist under
`personas/` and can be diff'd via the verifier.

---

## 7. Phase 1.2: LLM-generated personas (deferred)

When the LLM sampler is implemented, you'll add prompts under
`prompts/system_anchors.md`, `prompts/persona_rules.md`, and
`prompts/batch_guidance.md`. The orchestrator builds a system prompt from
the templates + your axes + joints + news context, and a per-batch user
prompt from the current gap report.

---

## 8. Phase 2: Run a simulation against a persona set

```bash
python simulations/up_2027_ac042/run.py experiments/vote_2027_baseline.yaml
```

The experiment config cites a persona set by name and defines the question
(direct vote ask, news-fed update, etc.). Output goes to
`runs/<run_name>/results.jsonl`.

## Common pitfalls

  - **CSV labels vs joint keys mismatched.** The build script must produce
    consistent canonical codes for the same conceptual entity (e.g., a caste
    leaf "Bagdi" should appear identically in axes.json categories and any
    joint that mentions Bagdi). Use a `CANONICAL_CODES` dict like in the WB
    build script.
  - **Marginal targets that include children** when the rest of the data is
    voter-only (or vice versa). Filter age cohorts in the build script and
    renormalize.
  - **Conditional axes** (occupation, class_of_worker) that target only a
    subset of the population (workers). Add `verify_condition` to the axis
    so the verifier matches the marginal's denominator.
  - **Joints with compound buckets** (e.g., age "33–47" instead of 5-yr bins)
    are handled by the SAMPLING side via plugins, but should be marked
    `use: "sampling_only"` so the verifier doesn't try to check them
    directly.
