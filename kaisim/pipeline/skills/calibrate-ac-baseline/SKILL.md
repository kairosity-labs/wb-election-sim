# Kaisim — Calibrate an AC's Baseline Population to Pass the Verifier

You are taking one constituency's auto-generated structures + plugins and
iteratively fixing them until `generate.py baseline_rule` reports a
composite chi-square ≤ 800.

This is a **mechanical curation job**, not a research job. The data is
already in the MD/CSVs; the auto-builder produces a starter that almost
works; the gaps are name-mismatches, missing aggregations, and a few
per-AC plugin tunings. About 90% of the work is on the checklist below.

## Reference

- AC 095 (`simulations/wb_2021_ac095/`) is the **fully hand-curated reference**.
  Its production population scores composite ≈ 359.
- AC 003 (`simulations/wb_2021_ac003/`) is the **first worked example** —
  see [`scripts/per_ac/003_curate.py`](../../../scripts/per_ac/003_curate.py)
  and [`scripts/per_ac/003_extensions.py`](../../../scripts/per_ac/003_extensions.py).
  It went 8842 → 611 by following this skill.

## Pre-flight check

Before starting, the AC needs:
- `data/calibrated_2019/<NN>_*_2019.md` (the source-of-truth markdown)
- All gates passing in `scripts/validate_calibrated_2019.py <NN>` (warn-only OK)
- `simulations/wb_2021_ac<NN>/structures/{axes,joints,aggregate_targets}.json`
  generated by `scripts/build_ac_verifier_configs.py <NN>`
- `simulations/wb_2021_ac<NN>/derived/` + `persona_configs/baseline_rule.yaml`
  + `generate.py` scaffolded by `scripts/scaffold_ac_baseline_runtime.py <NN>`

If anything is missing, run those scripts first. **Stop and verify** that the
underlying CSVs are sane; this skill only fixes downstream alignment.

## The checklist (in order of typical impact)

### A. **Joint cell |z| > 30 anywhere → flag/category-name mismatch**

If a joint cell shows huge gaps (target 80%, observed 0% or vice versa), the
joint's row keys or column keys don't match the axis's category codes.

**Diagnose:** open `axes.json` and `joints.json`, compare the joint's
`table` keys / value-cell keys against the axis's `categories` or `flags`.

**Common patterns:**
| Symptom | Fix |
|---|---|
| `asset_given_*` joint has `TV/Banking/LPG` but axis flags are `Television/Banking_access/LPG_clean_cooking_fuel` | Rename joint cell keys via `_rename_flags(table, {"TV":"Television", ...})` |
| `caste_given_gp` parent values are `Census_towns_cluster` but axis cats are `U1_Census_towns_cluster` | Drop the `U1_/U2_` prefix from axis cats (or add it to joint keys) |
| `migration_given_religion` has `WB_other_dist` but axis has `WB_other_district` | Rename child cats |
| `caste_given_gp` produces sub-codes (`Rajbanshi_SC`, `Other_SC`) absent from axis (which has `SC_total`) | Aggregate sub-codes onto leaves; for compound `OBC_Other_Hindu` split by ratio (e.g., 30:70 between OBC and Other_Hindu_middle) |

Write all of these into a `scripts/per_ac/<NN>_curate.py` file modeled on
[`003_curate.py`](../../../scripts/per_ac/003_curate.py).

### B. **Axis "Non_worker" or "Never_married" target gap > 10pp → drop kids + tune housewife**

The auto-built `age_cohort` axis includes children (0_4, 5_9, 10_14, 15_17)
which together force ~30% of the population to Non_worker — but the
workforce target marginal is for adults only (≈ 36%).

**Fix in curate.py:**

```python
CHILD_AGE_CODES = {"0_4", "5_9", "10_14", "15_17"}

def _drop_children_renormalize(axis):
    cats = axis["categories"]
    keep = [c for c in cats if c not in CHILD_AGE_CODES]
    kept_mass = sum(axis["marginal"].get(c, 0) for c in keep)
    axis["categories"] = keep
    axis["marginal"] = {c: axis["marginal"][c] * 100 / kept_mass for c in keep}
```

If after dropping kids you still have Non_worker > target by 5pp+, the
**female-housewife rate** in the workforce plugin is too aggressive.
The generic plugin uses 75% (matching AC 095 / WB rural FLFPR ≈ 23%).
For ACs with higher FLFPR (Cooch Behar / N. Bengal / tea-belt), drop
to 0–25%. Override in `simulations/wb_2021_ac<NN>/derived/workforce_sampling.py`:

```python
# AC-specific override — tune to hit Non_worker target
if gender == "Female" and female_housewife_age and rng.random() < 0.0:
    return _pick_category(axis, ("Non_worker",))
```

### C. **Cells under "% of workers" axes look broken**

The verifier compares persona occupation distribution against the marginal
target. But occupation is only populated for workers (~60% of pop). Target
"32% Agricultural_labourer" is "% of workers" — the verifier counts "% of
all personas" and reports a huge gap.

**Fix:** add `verify_condition` to the axis in curate.py:

```python
if a["name"] in ("occupation", "class_of_worker"):
    a["verify_condition"] = {
        "field": "workforce_status",
        "values": ["Main_worker", "Marginal_worker"],
    }
```

### D. **Joint with compound age keys (`33_47_compound`, `48_62_compound`) shows |z|>10**

`married_given_age_gender` and `education_given_age_gender` from D.7/D.6
sometimes use compound buckets. The persona's age_cohort is a 5-year
cohort like `33_37`, which never matches `33_47_compound`.

**Fix:** expand the compound key to constituent 5-year cohorts in curate.py:

```python
if j["name"] == "married_given_age_gender":
    j["table"] = _expand_age_buckets(j["table"], {
        "18_22": ["18_22"], "23_27": ["23_27"], "28_32": ["28_32"],
        "33_47": ["33_37", "38_42", "43_47"],
        "48_62": ["48_52", "53_57", "58_62"],
        "63": ["63_67", "68"],
    })
```

### E. **"D.2 sub-castes treated as religions" symptom**

If `caste_given_religion` has parent values like `SC_Rajbanshi`,
`SC_Namasudra`, `UC_bhadralok` instead of `Hindu/Muslim/Other`, the
auto-builder mistreated the D.2 table. This should be caught by
`build_ac_verifier_configs.py`'s special-case handler. If not:

**Fix:** ensure builder is using `_build_caste_given_religion()` for D.2
(it should). If still broken, manually rewrite the joint in curate.py
with `Hindu/Muslim/Other` as parent keys, Hindu row containing the
sub-caste→leaf-aggregated distribution.

### F. **Joint with non-existent parent axis → `parent axis 'X' doesn't exist`**

`bilingual_given_media` (parent: `media_tier`) and `vote_given_welfare`
(parent: `welfare_exposure`) reference axes that don't exist in non-095
ACs.

**Fix:** drop these joints from `joints.json` in curate.py:

```python
if j["name"] in ("bilingual_given_media", "vote_given_welfare"):
    continue  # skip — parent axis not declared for this AC
```

(Or implement those axes per-AC if you actually need them for vote
blending. Most baselines don't.)

### G. **"Widowed" target 8% but observed < 2% → broaden marital plugin widow rates**

The generic marital plugin emits Widowed only at fairly tight age/gender
gates. AC-level widow rates depend on local demographics.

**Fix:** write an AC-specific `simulations/wb_2021_ac<NN>/derived/marital_sampling.py`
that broadens the widow rates per local demographic. AC 003 example:

```python
if age_60plus and gender == "Female" and rng.random() < 0.85:
    return _pick_category(axis, ("Widowed",))
```

### H. **"Student" target 11% but observed < 5% → boost student rule in workforce plugin**

Default student rule fires only for 18-22 with secondary+ education at
55%. To hit 11% population Student, you typically need 80–90% rate at
18-22, plus a smaller rate (15–20%) for 23-27 graduate-or-above.

### I. **Vote-blending joint hygiene** (added from AC 011, 159, 210)

Even after every joint cell aligns, the `vote_2019_LS_share` aggregate-
target fails — or one party (often INC, BSP, or NOTA) has |z| > 5 because
its share was miscalibrated.

**Three sub-patterns:**

**I.1 Party-name normalization across vote_given_*.** If `vote_given_religion`
uses `Other_NOTA` but `vote_given_caste` uses `Other`, the blender unions
parties → 6-party space → residual mass mis-allocated.

```python
for j in joints:
    if j["name"].startswith("vote_given_"):
        for parent, row in j["table"].items():
            if isinstance(row, dict) and "Other" in row and "Other_NOTA" not in row:
                row["Other_NOTA"] = row.pop("Other")
        j["parties"] = ["BJP", "AITC", "INC", "LF", "Other_NOTA"]
# Match in persona_configs/baseline_rule.yaml: parties: [BJP, AITC, INC, LF, Other_NOTA]
```

**I.2 Aggregate-target bucket splitting.** When the target has separate
`Others_X` and `NOTA` buckets but the joint lumps them as `Others_NOTA`,
no persona ever lands in either bucket → both reported as
`skipped_buckets`. Split joint cells by target ratio:

```python
OTHERS_SHARE = 0.51  # Others_X share of (Others_X + NOTA) per ECI Form-20
resid = row.pop("Others_NOTA", 0.0)
row["Others_X"] = resid * OTHERS_SHARE
row["NOTA"] = resid * (1 - OTHERS_SHARE)
```

**I.3 Macro reconciliation for tiny-share parties.** Vote-conditional joints
often carry CSDS *state-level* rates (INC=3% Hindu / 8% Muslim) that don't
reconcile to AC-level macro (INC=0.86% in Nandigram). The blender feeds
these straight through and over-shoots. Clamp small-share parties:

```python
for caste_key, row in j["table"].items():
    inc = row.get("INC", 0)
    if inc > 1.5 and target_macro_inc < 1.5:    # AC has near-zero INC
        excess = inc - 1.0
        row["INC"] = 1.0
        row["AITC"] = row.get("AITC", 0) + excess  # redistribute to dominant party
```

For `vote_given_religion` specifically, you can back-solve the Hindu/
Muslim rows directly from the macro target (e.g., AITC 63.14, BJP 30.09,
Hindu 73.5%, Muslim 26.4% → P(AITC|Hindu) = (63.14 − 0.264·82) / 0.735 = 56.5).
This single change drove AC 210 from composite 1208 → 707 in one iteration.

### J. **Multi-religion / sub-population conditional joint expansion** (added from AC 064, 159, 222)

Source CSVs encode 3-religion buckets (Hindu/Muslim/Other) but the AC's
religion axis has 4-5 categories (Hindu/Muslim/Sarna_ORP/Christian/Other).
Auto-builder collapses Sarna into "Other" → personas of religion=Sarna_ORP
get no caste/lang/asset draw and fall back to defaults. Symptom:
specific religion category at observed=0% vs target>5%.

**J.1 Multi-religion expansion** (Sarna, Jain, Sikh as 4th+ religions).
Rebuild every religion-keyed joint to enumerate ALL religion rows:

```python
# Solve P(child | new_religion) so weighted marginal hits target.
# Example for ST_total leaf with target 22.92, Sarna→ST 92, Hindu mass 84%:
#   P(ST | Hindu) = (22.92 − 0.92·10.9) / 84 = 12.4
new_table["Hindu"] = {"ST_total": 12.4, "SC_total": 23.2, ...}
new_table["Sarna_ORP"] = {"ST_total": 92.0, ...}
new_table["Other_not_stated"] = {"Christian_Other": 100.0}
```

Don't forget `religion_given_gp` — every gp-row needs a cell for every
axis religion, otherwise the conditional sampler can't emit it.

**J.2 Hindu-minority AC** (Murshidabad / Malda). When Hindu < 50% AND
the MD §C.2 sub-caste rows partly drop during `_map_subcat_to_axis_leaf`,
the Hindu row sums to < 100%. Rebuild from MD §C.2 % of total values
divided by religion marginal:

```python
hindu_dist = {
    "SC_total":           19.5 / 56.5 * 100,    # leaf_pct_of_total / hindu_share_of_total × 100
    "ST_total":            2.2 / 56.5 * 100,
    "UC_bhadralok":        5.5 / 56.5 * 100,
    "OBC_Hindu":           8.0 / 56.5 * 100,
    "Other_Hindu_middle": 21.4 / 56.5 * 100,
}  # sums to 100
```

**J.3 Muslim sub-caste explosion.** When the caste axis has K>1 Muslim
leaves (Sheikh / Ansari-Jolaha / Syed / OBC-Muslim), the auto-builder
puts 100% of Muslim mass into the FIRST lexical match. Declare per-AC
distribution:

```python
MUSLIM_SUBCASTE_SHARES = {
    "Muslim_Sheikh":              48.0,
    "Muslim_Ansari_Jolaha":       30.0,
    "Muslim_Syed_Pathan_Mughal":   8.0,
    "Muslim_other_OBC_Muslim":    14.0,
}
j["table"]["Muslim"] = dict(MUSLIM_SUBCASTE_SHARES)
```

**J.4 Bengali / Non-Bengali Hindu collapse** (Bhabanipur, urban Kolkata).
When source CSVs split Hindu into "Bengali Hindu" and "Non-Bengali Hindu"
sub-rows but the religion axis has only one `Hindu`, population-share-
weight the two sub-rows:

```python
W_BENG_HINDU = 0.578   # Bengali Hindu mass / Hindu total (compute from caste)
W_NB_HINDU   = 0.422
new_hindu_row = {k: row_beng.get(k, 0)*W_BENG_HINDU
                  + row_nb.get(k, 0)*W_NB_HINDU
                 for k in set(row_beng) | set(row_nb)}
```

### K. **Per-AC sampling architecture: caste from religion, not GP** (added from AC 011, 198)

**K.1 The two-conditional consistency problem.** The DAG samples `caste`
from ONE joint (typically `caste_given_religion`). The other joint
(`caste_given_gp`) becomes verifier-only — but its chisq stays high
(50-300+) because chained `Σ_religion P(caste|religion)·P(religion|gp)`
doesn't equal the directly-quoted `caste_given_gp` table.

**Fix A** (lighter): mark `caste_given_gp` as `use: "verifier_only"` AND
hand-rebuild its values to chained P(.) so the verifier's expected and
observed agree. AC 064 used this.

**Fix B** (stronger, AC 198 worked example): write a per-AC
`derived/caste_sampling.py` plugin that conditions caste on BOTH
parents. Wire it in baseline_rule.yaml:

```yaml
caste:
  method:   plugin
  module:   derived.caste_sampling
  function: sample
  after:    [gp_location, religion]
```

Plugin sketches:

```python
HINDU_CASTES = ("UC_bhadralok", "OBC_Hindu", "Other_Hindu_middle",
                "SC_total", "ST_total")
def sample(rng, ctx, axis, persona, persona_config=None, **kwargs):
    religion = persona.get("religion")
    if religion == "Muslim":
        return _pick_category(axis, ("Muslim",))
    if religion in ("Sarna_ORP", "Christian", "Other"):
        return _pick_category(axis, ("Christian_Sarna_only_Other",))
    # Hindu — use caste_given_gp restricted to Hindu sub-castes
    joint = _try_joint(ctx, "caste_given_gp")
    row = joint.table.get(persona.get("gp_location"), {})
    cats = [c for c in HINDU_CASTES if c in row and row[c] > 0]
    weights = [row[c] for c in cats]
    return rng.choices(cats, weights=weights, k=1)[0]
```

**K.2 Vote-blending consistency.** When the persona's caste was sampled
without conditioning on religion, the vote blender mixes
`vote_given_religion[Muslim]` (BJP=10) with `vote_given_caste[ST_total]`
(BJP=52) → Muslim personas vote BJP at 50%. **Sample caste FROM
`caste_given_religion` first, not gp.**

## Extension to section B (workforce / housewife) — local FLFPR variations

The 75% generic female-housewife rate is calibrated to all-WB FLFPR ≈ 23%.
Many ACs deviate **either way**:

| AC archetype | FLFPR direction | Rate to use |
|---|---|---|
| WB rural baseline (matches 095) | ~23% | 75% (default) |
| Tea-belt (Kalchini, Dooars) | women dominate workforce | **0–5%** |
| Tribal-belt (Jhargram, Jangalmahal) | forest-produce + MGNREGA | **0–15%** |
| Cooch Behar / N. Bengal ag-belt | ~32% FLFPR | **40–55%** |
| Murshidabad Muslim-rural | conservative cultural | **15–25%** |
| Urban-Kolkata white-collar (Bhabanipur) | educated working women | **40–50%** |

**Workforce residual reweighting** (AC 198, 222): when senior + housewife
gates additively pile into Non_worker, the residual marginal sampler
double-counts. Instead of marginal weights, use a hand-tuned weight
map for the residual:

```python
# In per-AC workforce_sampling.py
def _sample_residual(rng, axis):
    weights_map = {"Main_worker": 55, "Marginal_worker": 20,
                    "Unemployed": 12, "Non_worker": 13}
    cats = list(weights_map.keys())
    return rng.choices(cats, weights=[weights_map[c] for c in cats], k=1)[0]
```

Or extend `_sample_from_marginal_excluding(reweights={"Non_worker": 0.75})`
to scale Non_worker's marginal weight down by ~0.7-0.8 to compensate for
the additive senior gate contribution.

## Extension to section G (marital widow rates) — ever-married split

If `married_given_age_gender` joint values look high (90%+ for Female 33-47),
those are likely **ever-married** rates from MD §D.7, NOT
**currently-married**. The generic plugin treats them as currently-married
which over-counts because it never deducts widows.

**Fix:** in per-AC marital plugin, treat the joint as P(ever-married | age,
gender) and split via age-conditional widow probability:

```python
p_ever_married = float(row[gender_key]) / 100.0
p_widow_given_ever = (
    0.65 if (age_60plus and gender == "Female") else
    0.40 if (age_50plus and gender == "Female") else
    0.10 if age_40plus else 0.02
)
p_currently_married = p_ever_married * (1 - p_widow_given_ever - 0.02)  # 0.02 separated
p_widowed = p_ever_married * p_widow_given_ever
# Cumulative: r < CM → CM; r < CM+W → W; r < CM+W+S → S; else Never
```

## Extension to section F (drop joints with mismatched parents)

Add to the drop list when the AC has **AC-local workforce sub-categories**:

`workforce_given_education` semantics is `two_indicator_rates` and
hardcodes child value `Main_worker` / `Unemployed_educated_actively_seeking`.
If the AC's workforce axis splits Main_worker into local subcategories
(e.g. AC 011's `Main_worker_tea_garden_wage_labor` and `Main_worker_non_tea`),
the verifier's lookup `p.get("workforce_status") == "Main_worker"` returns
False and observed=0 in every cell.

```python
if j["name"] == "workforce_given_education":
    # Skip — AC has split Main_worker subcategories that don't match
    # the two_indicator_rates verifier semantics.
    continue
```

## Extension to section A (alignment) — Mahishya is GENERAL, not OBC

When the auto-builder produces `caste_given_gp` rows with combined
`Mahishya_Sadgop` codes in OBC-dominant Hooghly/Howrah ACs (AC 198,
similar pattern in 210), naive routing of all Mahishya_Sadgop mass to
`OBC_Hindu` produces 40+% OBC observed vs 30% target.

Mahishya is a GENERAL/UC-equivalent caste in the WB Mandal classification
(despite being numerically a "Sudra" caste historically). Sadgop IS OBC.
Split the combined code ~70:30:

```python
ms_total = row.get("Mahishya_Sadgop", 0)
obc_share = ms_total * 0.30   # → OBC_Hindu
ohm_share = ms_total * 0.70   # → Other_Hindu_middle
```

## Stopping criteria

- **Composite ≤ 800** → success. Commit and ship.
- **Composite < 1500 but cells |z| > 2.5** → "passing on whole-population
  test, failing on individual cells." Often acceptable for v0 (the cells
  are tail-end small-n issues). Document in the AC's notes.
- **Composite > 5000 after 5+ iterations** → an upstream MD-level data
  inconsistency. Re-read §C/§D of the AC's MD; verify the source CSVs
  are derivable. Fix the MD first, re-run `derive_calibrated_2019_csvs.py
  <NN>`, then come back to curate.

## When in doubt

Diff the broken AC's `axes.json` against AC 095's. Usually the answer is
visible: 095 has subgroup_rollups, bucket_maps, compact category codes,
and the joint table values use the same keys as the axis. Make the new
AC look like 095 — but with **its own** local categories, not 095's
Matua/Bangaon-specific ones.

The skill is mechanical, not analytical. **Don't over-think it; just
work the checklist.**
